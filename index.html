Everything outside the day job's OpenGL ES map renderer lives here: a 4-thread engine (World · Render · RHI · Asset) that mirrors Unreal's separation of game logic from rendering, built on top of the four systems below — none of them from a library.

RenderCommandList is zero-allocation: two LinearArenas ping-pong — the World thread writes RenderTask lambdas into one while the Render thread drains the other, so a frame boundary never costs a heap allocation.

Four layers, four class diagrams

World owns game logic and never imports a rendering type. Renderer mirrors World's state one frame behind. RHI is the only layer allowed to call a graphics API. Asset feeds both, off the main threads.

MeshDrawCommand & RenderPass, specifically

MeshBatch owns exactly one MeshDrawCommand per (mesh, material) pair.
It carries two SSBOs — rawTransform (raw position / quaternion / scale per instance) and transform (the computed mat4s TransformPass writes) — plus the vertex layout and a material reference.
GeometryPass reads one MeshDrawCommand, issues one draw call.
Bind vertex layout, bind the transform SSBO, bind material — a single indexed draw covers every instance in the batch. This is the mechanism behind the "1 draw call per batch" result below.
RenderPass splits into two abstract bases, not one.
ComputePass just dispatches a compute shader — used by TransformPass (GPU TRS) and CullingPass (currently a pass-through stub). GraphicPass owns full pipeline state — clear, depth, blend, rasterizer — used by GeometryPass and LightingPass.
RenderGraph is just an ordered list of RenderPass.
Execute() walks it once a frame: TransformPass → GeometryPass → LightingPass ×N lights. Adding a pass means appending to the list, not touching the other passes.

What changed when it was measured

TransformPass moved TRS→matrix onto the GPU.
A compute shader (local_size_x=64) replaced per-instance CPU GLM math. 500K instances @ 20 FPS Release — roughly 10× over the CPU path, which couldn't hold 10 FPS Debug past 30K instances.
Hash Storm: an unconditional insert was the real bottleneck.
Every instance mutation queued a HashSet::Insert — 2,000,000 calls/frame at 1M instances, 15–20% of frame time. An if (!IsDirty()) guard cut that to once per frame regardless of mutation count: 5.4 → 22 FPS Debug, 185ms → 45ms/frame.
MeshBatch grouping turned N proxies into 1 draw call.
Same-mesh, same-material proxies share one MeshDrawCommand — see above.
Shadow shimmering — fixed by snapping, not by more samples.
Camera-frustum center quantized to the shadow map's texel grid before recomputing the light view matrix, using a bounding-sphere frustum. Removes shadow jitter under camera motion without extra PCF taps.

ECSown repository

github.com/Winteradio/ECS-with-Scene-and-Node

Entities are names; Components hold data with no logic; Nodes filter entities down to the exact component combination a System needs — Systems never scan raw entities. Execution order between Systems isn't declared by hand: each System calls DependOn(other), and a dependency graph topologically sorts the whole set before every update, detecting cycles as a build-time-cheap safety net rather than a manual ordering list that silently rots.

A trap I documented in the README, not just fixed once: the GC only sees a member variable if it's declared with PROPERTY(). Forget it on an EntityContainer or SystemRegistry member and the collector treats it as unreachable — it gets freed out from under you on the next Memory::Collect(), and the failure shows up nowhere near the missing macro.

Memoryown repository

github.com/Winteradio/MemoryPool

A Mark-and-Sweep collector with an incremental budget: Mark and Sweep each carry a TimeLimit, so a collection cycle can suspend mid-pass instead of stalling a frame. Objects are handed out through ObjectPtr<T>; RootPtr<T> additionally registers itself with the collector as a GC root on construction. Underneath, a StorageManager routes each type to a PoolBudget (single objects) or ArrayBudget (array objects), each classified by occupancy — Sparse / Moderate / Dense / Full — so allocation doesn't walk a pool that's already packed.

Robin Hood hashing swaps a newly-inserted key with whichever occupant has a lower probe-sequence-length, so no single key ever ends up stranded far from its ideal slot — variance stays low without chaining.
#25 Use-After-Free in DynamicArray Move Constructor closed
The move constructor moved m_data but not m_allocator — the moved-from object kept a live, default-constructed allocator with no real authority over the transferred memory, so its destructor freed memory it no longer owned. Fix: move the allocator alongside the data in the initializer list, same fix ported to move-assignment.
#24 HashMap::operator[] Reference Invalidation during Rehash open
m_map[A] = m_map[B] can crash: the RHS reference is taken first, then the LHS access triggers a rehash that relocates and frees the old backing array — the assignment then reads through a dangling reference. Doesn't happen on std::unordered_map (node-based, stable addresses); does happen here because contiguous open addressing is exactly what makes lookups cache-friendly. Logged as an inherent trade-off of the design, not patched over.

Reflectionown repository

github.com/Winteradio/Reflection

Three macros — GENERATE, PROPERTY, METHOD — register a class's type, member variables, and member functions at compile time via C++17 inline static members, so the definitions can live entirely in the header with no linker errors. At runtime, string-name lookups return PropertyInfo/MethodInfo instead of requiring dynamic_cast or hand-written switch statements.

Zero-allocation type erasure: a virtual-function iterator (IIterator*) would new on every container traversal. Instead the iterator carries a fixed 64-byte buffer (alignas(void*) uint8_t[64]), placement-new's the real STL iterator into it, and dispatches through captured lambdas instead of a vtable — copy/destroy are explicit deep-copy functions so Debug mode never sees a double-free from a shallow buffer copy.
Static method objects without new: two different member functions with identical signatures would collide on the same static storage if keyed only by type. MethodCreator<Func, func> keys on the function pointer value as a non-type template parameter, so each function gets its own template instantiation — and therefore its own unique static MethodCall — for free.
