Memory Pool & GC

A Mark-and-Sweep collector referencing Unreal Engine's garbage collector, with an incremental time budget so a collection cycle can suspend mid-pass instead of stalling a frame.

01

Collector, Storage, Smart Pointers

UE-referenced Mark-and-Sweep, incremental

Mark and Sweep each carry a TimeLimit. Objects are handed out through ObjectPtr<T>; RootPtr<T> additionally registers itself with the collector as a GC root on construction. Underneath, a StorageManager routes each type to a PoolBudget (single objects) or ArrayBudget (array objects).

02

No Compaction — Occupancy-Ranked Buckets Instead

a deliberate trade-off, not a missing feature

A classic compacting GC moves live objects together to eliminate fragmentation — but that only stays cheap if there's a stable indirection layer (a handle table) between a pointer and the object, so moving the object doesn't invalidate anything already holding a reference to it.

Why compaction was ruled out here: Accessor lives inline in each instance's own header, not in a separate stable handle table. Moving an object during compaction moves its Accessor with it — so every live ObjectPtr / RootPtr already pointing at that instance would need to be found and rewritten (back-propagated) to the new address. Without a tracked list of referrers, that's effectively a second full GC pass on its own. Compaction was left out on purpose.
Fragmentation is managed on the allocation side instead: each PoolBudget classifies its buckets by occupancy ratio — eSparse / eModerate / eDense / eFull — and new allocations are routed to the bucket with the lowest occupancy first, rather than packing whichever bucket is already fullest. Spreading allocations this way keeps any single bucket from running away to eFull and forcing a fresh backing allocation, which is the practical source of internal fragmentation this design is actually trying to avoid.
