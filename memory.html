Memory Pool & GC

Unreal Engine의 가비지 컬렉터를 참고한 Mark-and-Sweep 방식이며, 사이클이 중간에 멈췄다가 이어서 재개될 수 있도록 시간 예산을 두고 있습니다.

01

핵심 내용

상태 하나가 전체 사이클을 이끕니다.
eIdle → eMarking → eSweeping → ePurging → eIdle 하나의 eStatus가 전부 결정합니다. 각 단계가 빨리 끝나면 Collect() 한 번 호출로 여러 단계를 연속으로 통과할 수 있습니다.
시간 예산은 정확히 Purge 단계에만 적용됩니다.
Mark와 Sweep은 항상 한 번에 끝까지 돕니다. 죽은 객체를 파괴하고 슬롯을 반환하는 Purge()만 객체 단위로 시간을 체크하고, 부족하면 중간에 멈췄다가 다음 Collect()에서 그 지점부터 이어갑니다.
Compaction은 실수로 빠진 게 아니라 의도적으로 뺐습니다.
Accessor(객체 메타데이터)가 별도 핸들 테이블이 아니라 각 인스턴스 자신의 헤더 바로 앞에 있어서, 객체를 옮기면 그 객체를 가리키는 모든 포인터를 찾아 다시 써야 합니다. 대신 버킷 점유율 기반 할당 정책으로 단편화를 관리합니다.

ObjectPtr vs. RootPtr

Root set은 "포인터를 가리키는 포인터"를 담습니다.
RootPtr의 생성자는 AddRoot(this)를 호출해서 관리 대상 객체의 주소가 아니라 RootPtr 래퍼 자신의 주소를 root set에 넣습니다. Mark()는 이 래퍼를 거쳐 실제 인스턴스에 도달합니다.
ObjectPtr은 비활성입니다.
레이아웃은 같지만 생성자/소멸자 어디서도 컬렉터와 대화하지 않는 평범한 핸들입니다. RootPtr로 감싸야만 GC root가 됩니다.

사용 예시

#include "Memory.h"
using namespace Memory;

class MyClass
{
    GENERATE(MyClass); // Reflection 시스템과 함께 쓰면 GC 순회 대상이 됨
public:
    int value = 42;
};

int main()
{
    Init(1024 * 1024, 1000000); // 풀 크기 1MB, Sweep 시간 예산 1ms

    ObjectPtr<MyClass> obj  = MakePtr<MyClass>();      // 비활성 핸들 — GC root 아님
    RootPtr<MyClass>   root = MakePtr<MyClass>();      // 생성자가 AddRoot(this) 호출 -> GC root
    RootPtr<int[]>     arr  = MakeArray<int>(5);

    obj = ObjectPtr<MyClass>();  // 참조 해제
    Collect();                    // obj가 가리키던 메모리는 회수 대상, root/arr는 보호됨
    Release();
}
03

Compaction을 하지 않은 이유

빠진 기능이 아니라 의도적인 트레이드오프
Compaction을 배제한 이유: Accessor는 별도의 안정된 핸들 테이블이 아니라 각 인스턴스 헤더에 인라인으로 존재합니다. Compaction 중 객체를 옮기면 그 Accessor도 함께 옮겨지므로, 이미 그 인스턴스를 가리키고 있던 모든 ObjectPtr/RootPtr 을 찾아서 다시 써야 합니다. 참조자를 추적하는 목록이 따로 없다면, 이건 사실상 GC를 한 번 더 도는 것과 같은 비용입니다.
대신 할당 시점에서 단편화를 관리합니다: 각 PoolBudget은 버킷을 점유율로 분류하고(eSparse / eModerate / eDense / eFull), 새 할당은 가장 꽉 찬 비어있지 않은 버킷부터 찾습니다 — 어느 한 버킷이 eFull까지 차서 새 백킹 할당을 강제하는 상황을 줄이고, sparse 버킷은 완전히 비워져 회수될 여지를 남겨둡니다.
04

결과물 / 이슈

#16 The multi-threading GC will be crashed 미해결
- 원인: AddRoot/RemoveRoot는 뮤텍스로 보호되지만, Accessor의 마킹 상태와 각 Pool의 used/free/dead 리스트는 그렇지 않습니다 — mutator 스레드가 ObjectPtr을 통해 Accessor 상태를 읽는 동안 컬렉터 스레드가 같은 Accessor를 건드리면 경합이 발생할 수 있습니다.
- 개선 방향: 기존 root set 뮤텍스 범위를 Accessor 상태와 pool 리스트 변경까지 확장하거나, 상태 필드를 atomic으로 바꾸는 방향.
