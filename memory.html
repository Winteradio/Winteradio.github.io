    participant App
    participant GC as GarbageCollector
    participant Root as m_rootSet

    App->>GC: Collect()
    GC->>GC: Prepare - copy every root onto a reusable DFS stack

    GC->>GC: Purge - destruct dead objects, time-limited

02

How It Actually Runs

verified against the current source, not just the design notes

The GC cycle

One status field, cascading phases
A single eStatus drives everything (eIdle → eMarking → eSweeping → ePurging → eIdle). A single Collect() call can fall through several phases in one invocation if each finishes fast.
Mark is an iterative DFS
Not recursive — roots are copied onto a reusable stack, then popped one at a time. Reflected properties are classified (container / struct / object) and any child BasePtrs get pushed back onto the same stack.
Sweep does double duty
Anything still eUnreachable (never marked this cycle) moves to a dead list. Everything else gets forcibly reset to eUnreachable right there — which is how "unmarked next cycle" is detected without a separate bit.
The time budget applies to Purge, specifically
Mark and Sweep always run to completion in one call. Only Purge() — destructing dead objects and returning their slots — checks the time limit per object and can bail mid-list, resuming from the same dead list next Collect() call.

ObjectPtr vs. RootPtr

The root set holds pointers to the pointers
RootPtr's constructor calls AddRoot(this) — inserting the RootPtr wrapper's own address into the collector's root set, not the managed object's address. Mark() dereferences through the wrapper to reach the real instance.
ObjectPtr is inert
Same layout, but no constructor/destructor ever talks to the collector — it's a plain handle. Only wrapping it in RootPtr makes something a GC root.
Confirms why compaction is off the table
The Accessor header sits immediately before the object in the same allocated chunk — there's no separate indirection table anywhere in this path. Moving the chunk would mean rewriting every ObjectPtr/RootPtr caching that raw address.

PoolBudget bucket selection

Four density buckets, real thresholds
eSparse [0, 0.3) · eModerate [0.3, 0.7) · eDense [0.7, 1.0) · eFull [1.0, +), computed from used/total ratio.
Allocation prefers the fullest non-full pool
Search order is Dense → Moderate → Sparse, returning the first non-empty bucket found. A brand-new backing pool is only created once all three are empty. This keeps sparse pools available to potentially empty out completely, rather than spreading new allocations thin across many partially-used pools.
Reclaiming
Only pools that have drifted down to eSparse and are fully empty get freed back to the OS — a pool that cycles through Dense/Full is never released until it works its way back down.
Why compaction is out, in one line: the Accessor lives inline in each object's own header rather than a stable handle table, so moving an object during compaction would mean finding and rewriting every live pointer to it — without a tracked referrer list, that's effectively a second full GC pass.

Real Issue

#16 The multi-threading GC will be crashed open
- Cause: AddRoot/RemoveRoot are mutex-guarded, but the Accessor's mark-status flags and each Pool's used/free/dead lists are not — a mutator thread reading an Accessor's status through ObjectPtr can race with the collector thread's Mark/Sweep touching the same Accessor.
- Direction: extend the existing root-set mutex (or move to atomics on the status field) to cover Accessor state and pool list mutation, not just root registration.
