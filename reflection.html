01

핵심 내용

단일 상속 체인만 지원하고, 그래서 Cast<T>가 안전합니다.
TypeInfo는 클래스당 SuperType 하나만 모델링합니다. 위로 올라가는 경로가 하나뿐이라 Cast<T>는 DAG 탐색이 아니라 그냥 체인을 걷는 것으로 끝납니다 — dynamic_cast가 전혀 필요 없습니다.
컨테이너 순회는 힙 할당이 0입니다.
Iterator가 64바이트 고정 버퍼에 실제 STL/wtr 반복자를 placement-new로 생성하고, vtable 대신 함수 포인터 6개로 디스패치합니다.
같은 시그니처의 함수 두 개도 서로 다른 static 객체를 가집니다.
MethodCreator<Func, func>가 함수 포인터 "값"을 비타입 템플릿 파라미터로 받기 때문에, 시그니처가 같아도 함수마다 별도 템플릿 인스턴스 → 별도 static이 생깁니다.

03

매크로가 실제로 확장되는 방식

GENERATE(ClassName)

SuperType 탐지
Utils::TypeDetector의 SFINAE로 베이스의 리플렉션 태그를 찾아 SuperType으로 alias합니다. ThisType은 클래스 자신을 가리킵니다.
GetTypeInfo() 오버라이드
함수 지역 static const TypeInfo*("magic static", 스레드 안전, 최초 1회만 초기화)를 오버라이드된 virtual이 반환합니다 — 베이스 포인터가 런타임에 실제 파생 타입을 복원하는 방법입니다.
자체 등록
inline static const TypeInfo* s_typeInfo = GetStaticTypeInfo(); — 정적 초기화 시점에 동적 초기자가 실행되고, TypeInfo 생성자 마지막에서 TypeManager::Regist(this)를 호출해 unordered_map 에 등록합니다. TypeManager::GetHandle()이 Meyer's singleton이라 TU 간 정적 초기화 순서 문제에서 자유롭습니다.
이름 붙은 링커 위험 요소
__STATIC_USED__(GCC/Clang의 __attribute__((used))) 가 "참조되지 않는" static이 스트리핑되는 걸 막으려고 존재하는데, 늘 이기는 건 아닙니다(아래 이슈 참고).

PROPERTY(member)

오프셋 캡처
offset = reinterpret_cast<size_t>(&(static_cast<Class*>(nullptr)->*pointer)) — null 베이스 포인터를 통해 멤버 포인터를 역참조합니다. vtable이 있는(non-standard-layout) 클래스에서는 offsetof가 지원되지 않아 필요한 방식입니다.
Get/Set
둘 다 reinterpret_cast<char*>(instance) + m_propertyOffset 으로 해석됩니다. 대입은 캡처 없는 람다가 void(*)(void*, const void*) 썽크로 decay되어 *(Property*)dest = *(const Property*)src를 수행합니다.

Zero-allocation 컨테이너 Iterator

vtable도 베이스 클래스도 없음
alignas(void*) uint8_t m_storage[64]에 함수 포인터 6개 (Next/Prev/Destroy/Element/Copy/Compare) — 컨테이너 타입별로 고정된 이 함수 포인터 묶음 자체가 수동으로 만든 vtable입니다.
생성
begin()이 실제 STL/wtr const_iterator를 64바이트 버퍼에 placement-new로 직접 생성합니다.
복사는 진짜 깊은 복사입니다
버퍼를 그냥 memcpy하면 내부 포인터(해시 버킷, 리스트 노드)를 들고 있는 반복자가 깨집니다. CopyFunc가 원본에서 새로 placement-new합니다.
소멸은 명시적입니다
uint8_t[]는 소멸자를 자동으로 안 불러줍니다 — ~Iterator()가 DestroyFunc를 호출해 그 자리에서 실제 반복자의 소멸자를 실행합니다.
04

결과물 / 이슈

PropertyInfo missing in template classes on GCC/Clang 미해결
- 원인: PROPERTY()가 템플릿 클래스 내부에 만드는 static 객체가, 템플릿이 겉보기에 안 쓰이는 것처럼 보이면 GCC/Clang의 공격적인 dead-code elimination에 의해 조용히 스트리핑되어 프로퍼티/GC 등록이 깨집니다.
- 개선 방향: 컴파일러 used/retain 속성으로 강제 보존 — __STATIC_USED__가 노리는 것과 같은 종류의 해법이지만 아직 템플릿까지는 완전히 커버하지 못했습니다.
TypeInfo initialization failure with forward declarations 해결
- 원인: TypeInfo<T>는 메타데이터 추출을 위해 컴파일 타임에 완전한 타입의 크기/정렬 정보가 필요한데, 전방 선언만으로는 이걸 제공할 수 없습니다.
