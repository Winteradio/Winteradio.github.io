02

How It Actually Runs

what the macros expand to, mechanically

GENERATE(ClassName)

SuperType detection
SFINAE via Utils::TypeDetector finds a base's reflection tag and aliases it as SuperType; ThisType aliases the class itself.
GetTypeInfo() override
A function-local static const TypeInfo* (a "magic static," thread-safe, initialized once) is returned by an overridden virtual — this is how a base pointer recovers the real derived type at runtime.
Self-registration
inline static const TypeInfo* s_typeInfo = GetStaticTypeInfo(); — its dynamic initializer runs at static-init time, and TypeInfo's constructor ends by calling TypeManager::Regist(this), inserting into an unordered_map<hash, TypeInfo*>. Because TypeManager::GetHandle() is itself a Meyer's singleton, cross-TU static-init order never matters.
A named linker hazard
__STATIC_USED__ (__attribute__((used)) on GCC/Clang) exists specifically to stop the linker from stripping this "unreferenced" static — and it doesn't always win (see Issue #25 below).

PROPERTY(member)

Offset capture
offset = reinterpret_cast<size_t>(&(static_cast<Class*>(nullptr)->*pointer)) — dereferencing a pointer-to-member through a null base pointer. Needed because plain offsetof is unsupported/UB once a class has a vtable (non-standard-layout).
Get/Set
Both resolve as reinterpret_cast<char*>(instance) + m_propertyOffset. Assignment goes through a captureless lambda that decays to a plain void(*)(void*, const void*) thunk doing *(Property*)dest = *(const Property*)src.

The zero-allocation container Iterator

No vtable, no base class
alignas(void*) uint8_t m_storage[64] plus 6 plain function pointers (Next/Prev/Destroy/Element/Copy/Compare) — that fixed bundle of function pointers per container type is the manual vtable.
Construction
begin() placement-news the real STL/wtr const_iterator directly into the 64-byte buffer: new (&storage) ContainerConstIterator(itr).
Copy is a real deep copy
A raw memcpy of the buffer would corrupt any iterator holding internal pointers (hash-bucket, list node). CopyFunc placement-news a fresh iterator from the source instead.
Destruction is explicit
A raw uint8_t[] never runs destructors on its own — ~Iterator() calls DestroyFunc, which calls the real iterator's destructor in place.
One sharp edge
There's no static_assert(sizeof(ContainerConstIterator) <= 64) anywhere — an oversized concrete iterator type would silently overflow the buffer.

Cast<T>()

Compile-time fast path
If IsSame<T,U> or IsChild<T,U> (essentially std::is_base_of) already proves it safe, it's a plain static_cast — zero runtime cost.
Runtime path
Otherwise walks TypeInfo::m_superType one link at a time until it matches or hits null. Because that's a single pointer, not a set of bases, this is a linear walk — never a DAG search like dynamic_cast has to do.

MethodCreator<Func, func>

The trick
func is bound as a non-type template parameter — a value, not just a type. Two member functions with identical signatures (Foo(), Bar()) instantiate different specializations because the values differ.
The payoff
Each instantiation gets its own function-local static MethodCall — a unique, heap-free, per-function object, keyed by template identity instead of a runtime map.

Real Issues

#25 PropertyInfo missing in template classes on GCC/Clang open
- Cause: the static objects PROPERTY() creates inside template classes get dead-code-stripped by aggressive GCC/Clang linkers when the template looks otherwise unused, silently breaking property and GC registration.
- Direction: force retention with compiler used/retain attributes — the same class of fix __STATIC_USED__ targets, not yet extended to cover templates.
#29 TypeInfo initialization failure with forward declarations closed
- Cause: TypeInfo<T> needs a complete type's size/alignment at compile time to extract metadata — a forward declaration can't provide that.
