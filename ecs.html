ECS with Scene and Node

Entities are names; Components hold data with no logic; Nodes filter entities down to the exact component combination a System needs. A dependency graph — not a hand-written order — decides which System runs before which.

github.com/Winteradio/ECS-with-Scene-and-Node

Class Relationships

Systems never scan raw entities. Execution order between Systems isn't a hand-written list either — each System declares DependOn(other), and a graph topologically sorts the whole set before every update, catching cycles as a cheap safety net instead of a manual ordering that silently rots.

02

Node-filtered, not Sparse Set or Archetype

why the storage model isn't EnTT- or DOTS-shaped

The two dominant modern ECS storage strategies weren't the fit here — for reasons specific to what this engine actually needed from its ECS.

StrategyWhat it optimizes forWhy it didn't fit here
Sparse Set
(EnTT-style)		 O(1) add/remove, dense cache-friendly iteration — per single component type The moment a System needs several components together, it has to pick the smallest set and probe the rest at query time. This engine solves that once, at Node-registration time, instead: a Node is the multi-component filter.
Archetype
(Unity DOTS, Flecs)		 Best cache locality while an entity's component signature stays fixed Adding or removing a component migrates the entity's data to a different archetype's storage — a memory copy plus archetype-graph bookkeeping. This ECS is also where reflection-driven GC tracking lives (PROPERTY-registered members); migrating storage would mean re-establishing that tracking on every add/remove too.
Node-filtered
(this project)		 Simple, correct multi-component queries; no entity migration A Node is declared once per System; NodeContainer hands the System exactly the filtered view it asked for. No archetype graph, no per-query set intersection.

draft comparison — written from general Sparse-Set/Archetype trade-offs, not yet checked against the original research notes. Flag anything that doesn't match.

03

A Trap Worth Documenting

The GC only sees a member variable if it's declared with PROPERTY(). Forget it on an EntityContainer or SystemRegistry member and the collector treats it as unreachable — it gets freed out from under you on the next Memory::Collect(), and the failure shows up nowhere near the missing macro. This is in the README, not just fixed once and forgotten.
