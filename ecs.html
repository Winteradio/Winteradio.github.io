ECS with Scene and Node

Entities are names; Components hold data with no logic; Nodes filter entities down to the exact component combination a System needs. A dependency graph — not a hand-written order — decides which System runs before which.

Class Relationships

Systems never scan raw entities. Execution order between Systems isn't a hand-written list either — each System declares DependOn(other), and a graph topologically sorts the whole set before every update, catching cycles as a cheap safety net instead of a manual ordering that silently rots.

02

Node: a deliberate middle point between Sparse Set and Archetype

from the original research notes, before this repo existed

Both mainstream ECS storage strategies were studied first — starting from Unity DOTS's data-oriented pitch (same-type data contiguous in memory, systems operating on data instead of objects) — and both had one specific failure mode that mattered for this engine.

StrategyHow it worksThe specific problem
Sparse Set
(EnTT-style)		 A dense set holds Entity IDs, a sparse set holds components — fast contiguous-memory component access by ID. Access is scoped to one component type via the Entity ID. Not every System can be expressed against a single component.
Archetype
(Unity ECS)		 Entities with the same component set share an archetype; archetypes (not components) are what's laid out contiguously. Adding even one component to an entity means a new archetype — duplicate the shared components into it, remove them from the old one. Wanting an object that both emits light and renders means standing up a dedicated "Light Render Archetype."
Node borrows from both, to dodge both problems. A Node declares the exact component combination a System needs — closer to Archetype's "group by composition" idea — but a Node is attached to an entity without moving or duplicating that entity's component data anywhere, closer to Sparse Set's "just look it up by ID." Wanting an object that emits light and renders no longer means a new archetype: register both a RenderNode and a LightNode on the same entity, and both can read the same TransformComponent without either one owning or copying it.

source: personal research notes, Entity Component System — written before this repository existed; the shipped design (reflection-driven Cast instead of dynamic_cast, PROPERTY-tracked GC roots) is more advanced than the notes describe, but the Sparse-Set/Archetype reasoning above is unchanged from them.

A Trap Worth Documenting

The GC only sees a member variable if it's declared with PROPERTY(). Forget it on an EntityContainer or SystemRegistry member and the collector treats it as unreachable — it gets freed out from under you on the next Memory::Collect(), and the failure shows up nowhere near the missing macro. This is in the README, not just fixed once and forgotten.
