ECS with Scene and Node

Entity는 이름일 뿐이고, Component가 데이터를 들고, Node가 System에 필요한 정확한 컴포넌트 조합을 묶습니다. 실행 순서는 손으로 짠 목록이 아니라 의존성 그래프가 정합니다.

01

핵심 내용

Node 멤버십은 자동으로 검증되지 않습니다.
Node는 ObjectPtr<T> 필드로 구성된 평범한 struct이고, 생성 시점에 수동으로 연결합니다. 프레임워크가 Entity의 컴포넌트를 스캔해서 확인해주는 과정은 없고, System이 사용 시점에 방어적으로 null 체크를 하는 게 실질적인 안전장치입니다.
"필터링"은 사실 Node 타입별 HashMap 하나일 뿐입니다.
각 Node 타입은 자기만의 UUID → ObjectPtr<T> 저장소를 갖고, System은 그 컨테이너를 그대로 받습니다. 프레임마다 훑으면서 필터링하는 과정은 없습니다.
의존성 그래프는 교과서적인 칸의 알고리즘(Kahn's algorithm)입니다.
DependOn() 엣지로 진입 차수(in-degree) 맵을 만들고, 진입 차수 0인 System부터 큐에 넣어 위상정렬합니다. 순환이 있으면 조용히 실패 — 예외를 던지지 않고 로그만 남긴 뒤 빈 배열을 돌려줍니다.
02

상세 내용 & 다이어그램

실제 소스 코드 기준으로 검증
클래스 다이어그램 — Entity / Component / Node / System / Graph
시퀀스 다이어그램 — World::Update() 한 틱
GC 도달 가능성은 범용 프로퍼티 순회입니다.
클래스별로 따로 짠 추적 함수가 아니라, TypeInfo의 프로퍼티 목록을 그대로 따라갑니다. PROPERTY()로 등록하지 않은 ObjectPtr/RefPtr 멤버는 이 순회에 보이지 않습니다 — 아래 GC 함정이 예외적인 버그가 아니라 이 설계의 직접적인 결과인 이유입니다.
GC는 PROPERTY()로 선언된 멤버만 추적합니다. Registry 멤버에 이걸 빠뜨리면 컬렉터가 도달 불가능하다고 판단해서, 다음 Memory::Collect() 때 조용히 해제해버립니다.
03

Sparse Set도 Archetype도 아닌 Node

방식문제점
Sparse Set빠르지만 Entity ID로 접근 가능한 컴포넌트가 하나뿐 — 모든 System이 컴포넌트 하나로 환원되지는 않습니다.
Archetype컴포넌트 하나만 추가해도 새 archetype이 필요 — 기존 데이터를 복제해서 옮기고 원래 자리에서 지워야 합니다.
Node (이 프로젝트) Entity의 데이터를 옮기거나 복제하지 않고도 System이 필요한 컴포넌트 조합을 선언합니다. 라이트+렌더 = 같은 Entity에 두 Node를 등록하고 Transform을 공유합니다.

출처: 개인 리서치 노트, Entity Component System

- 템플릿 래퍼가 AddNode(nodeType) 대신 AddComponent(nodeType)를 호출합니다. 짝인 RemoveNode<T>()는 맞게 작성돼 있어서 더 눈에 안 띄는 비대칭 버그입니다. 이 경로로 등록한 Node는 HasNode<T>()가 항상 false를 반환합니다.
src/Object/Entity.cpp:47-59 Entity의 move 대입 연산자가 실제로는 아무것도 move하지 않음 버그 · 미수정
- Object::operator=(other)가 lvalue를 받아 베이스 클래스는 복사되고, m_name = std::move(m_name)은 other.m_name이 아니라 자기 자신을 self-move — 사실상 no-op입니다. "move" 대입인데 이름은 그대로, UUID만 원본과 같은 값이 됩니다.

include/Graph/Graph.h:18-28 GraphObserver::m_updated가 초기화되지 않음 버그 · 미수정
