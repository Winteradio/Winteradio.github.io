02

How It Actually Runs

verified against the current source
Node membership isn't automatically checked
A Node is a plain struct of ObjectPtr<T> fields, wired up by hand at creation: node.Create<RenderNode>(entityID, cResource, cTransform, cColor). Nothing in the framework scans an Entity's components to verify a Node's fields before handing it out — Systems defensively null-check each field before using it, which is the actual safety net today.
"Filtering" is really just one HashMap per Node type
Each Node type gets its own flat UUID → ObjectPtr<T> storage. A System asking for its Node type just gets handed that container directly — an entity only has an entry if a Node was explicitly created for it, so there's no per-frame scan-and-filter happening.
The dependency graph is a textbook Kahn's algorithm
In-degree map built from every DependOn() edge, a queue seeded with in-degree-0 systems, each pop decrementing its dependents' in-degree.
Cycles fail silently, on purpose
If the sorted count doesn't match the total system count, a cycle exists — it's logged, not thrown. GetSorted() returns an empty array rather than crashing, and a dirty flag skips rebuilding the sort when nothing has changed since the last call.
GC reachability is a generic property walk
Not hand-written per-class trace functions — the collector walks whatever TypeInfo's property list says exists. Any ObjectPtr/RefPtr member without PROPERTY() is invisible to that walk, which is exactly why the GC trap below is a direct consequence of the design, not an edge case.
The GC only tracks a member declared with PROPERTY(). Miss it on a Registry member and the collector treats it as unreachable — freed out from under you on the next Memory::Collect().

Node vs. Sparse Set vs. Archetype

StrategyThe problem
Sparse SetFast, but access is scoped to one component type by Entity ID — not every System reduces to a single component.
ArchetypeAdding one component means a new archetype — duplicate the shared data in, remove it from the old one.
Node (this project) A Node declares the component combo a System needs, without moving or duplicating the entity's data. Light + render on one entity = register both Nodes, share the Transform.
