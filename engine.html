커스텀 C++ 게임 엔진

World · Render · RHI · Asset, 4개 스레드로 게임 로직과 렌더링을 분리한 Unreal 스타일 구조입니다. 본업의 OpenGL ES 지도 렌더러 바깥에서 하는 모든 것이 여기 있고, 상단 메뉴의 네 시스템 위에 얹혀 있습니다.

01

핵심 내용

이 엔진이 실제로 하는 일
World는 렌더 타입을 절대 참조하지 않습니다.
ProxyComponent가 IsDirty()만 뒤집으면, Scene이 감지해서 Commander를 통해 스냅샷을 내보냅니다. World 코드 어디에도 렌더러 타입이 등장하지 않습니다.
RHI만 그래픽스 API를 직접 호출합니다.
OpenGL 컨텍스트는 스레드 종속성이 있어서, 모든 GPU 호출은 커맨드 객체(실측 41개)로 감싸고 RHIWorker만 실행합니다.
RenderCommandList는 힙 할당이 거의 없습니다.
두 개의 LinearArena가 핑퐁 방식으로 동작 — World 스레드가 하나에 람다를 쓰는 동안 Render 스레드가 다른 하나를 비웁니다.
100만 인스턴스 최적화의 진짜 병목은 "무조건 실행되는 삽입"이었습니다.
Dirty 추적용 HashSet::Insert가 프레임당 200만 번 무조건 호출되던 것을 if (!IsDirty()) 가드 하나로 5.4 → 22 FPS까지 끌어올렸습니다.

03

네 개의 레이어 — 상세 내용 & 다이어그램

World는 렌더 타입을 모르고 · RHI만 그래픽스 API를 부릅니다

WorldContext

엔티티, 컴포넌트, 그리고 밖으로 나가는 단방향 다리입니다. Scene::Flush()는 추가/삭제/갱신 3단계를 순서대로 도는데, 각 단계가 끝날 때마다 대상 HashSet을 비웁니다.

ProxyComponent는 렌더러와 직접 대화하지 않습니다 — IsDirty()만 뒤집으면 Scene이 알아채고 Commander를 통해 스냅샷을 내보냅니다. 실제로 Commander가 캡처하는 람다는 [renderView](Renderer*, RefPtr<RHICommandList>){ renderer->SetView(renderView); } 처럼 값 캡처 위주라 실질적으로 힙 할당이 거의 없습니다.

Renderer

RenderScene과 RenderGraph를 소유합니다 — 렌더링 쪽 객체는 전부 이 둘 중 하나를 거칩니다.

(mesh, material) 조합 하나당 MeshDrawCommand 하나 → Draw Call 하나.
GeometryPass가 vertex layout + transform SSBO + 머티리얼을 바인딩하고, 배치 안의 모든 인스턴스를 인덱스 Draw 한 번으로 그립니다.
RenderPass는 ComputePass와 GraphicPass, 두 개의 추상 기반으로 나뉩니다.
ComputePass는 컴퓨트 셰이더 디스패치만 담당하고, GraphicPass는 clear/depth/blend/ rasterizer 같은 파이프라인 상태 전체를 소유합니다.
RenderGraph는 실제로는 그래프가 아니라 고정된 체인입니다.
Transform→Shadow→Geometry→Lighting→Debug 순서로 SetDependency가 미리 걸려 있어서, "그래프" 구조인데 실제 파이프라인은 분기 없는 직선입니다.
TransformPass 디스패치는 1차원 고정 — 64 인스턴스당 워크그룹 1개.
groupX = ceil(instanceCount / 64), Y/Z는 항상 1. transform.cs.glsl의 local_size_x = 64와 맞물려 있는데, 생성자에서 넘기는 ComputePass(64, 64, 64) 그룹 크기는 실제로는 아무 데서도 읽히지 않는 죽은 인자입니다 — 디스패치 시점에 로컬 상수 64를 다시 씁니다.
CullingPass는 스텁이지만 배선은 이미 되어 있습니다.
Draw()는 상태 체크만 하고 항상 true, GetResourceState()도 항상 eReady로 하드코딩. 다만 카메라/인다이렉트 드로우 버퍼/가시성 버퍼/AABB 버퍼를 바인딩하는 SetCommand()는 이미 존재합니다. 짝을 이루는 culling.cs.glsl은 이름과 달리 실제로는 compute가 아니라 vertex 셰이더 내용이라, 컬링은 셰이더·디스패치·패스 로직 전부 미구현 상태입니다.
OverlayPass는 완전히 빈 파일입니다.
OverlayPass.h/.cpp 둘 다 0바이트 — 클래스 선언조차 없는, 시작조차 안 된 자리표시자입니다.

RHI

그래픽스 API를 직접 호출할 수 있는 유일한 레이어입니다. 실제 커맨드 클래스는 41개(초기화 11 · 갱신 3 · 리사이즈 2 · 제거 7 · Set 8 · Unset 7 · 프레임/디스패치 3)이고, 전부 CRTP로 만든 RHICommand<T> 라 실행 시점엔 vtable이 아니라 함수 포인터 하나로 디스패치됩니다.

OpenGL 컨텍스트는 스레드 종속성이 있어서, 다른 스레드에서 호출하면 정의되지 않은 동작이 됩니다. 모든 GPU 호출은 커맨드 객체로 감싸고 RHIWorker만 실행합니다.

Vulkan 백엔드 진행 중

기존 GLSystem과 같은 RHISystem 인터페이스를 구현하는 VulkanSystem으로 붙습니다 — 별도 아키텍처가 아니라, 지금 있는 RHICommand<T> 커맨드 시스템과 RHIWorker 스레딩 모델 위에 그대로 얹힙니다.

여기까지 실제로 동작합니다: 초기화 + 프레임 루프.
Instance/Device 선택, Surface/Swapchain 생성, Primary/Secondary 커맨드 버퍼 분리 (RHI 스레드용 vs 워커별), BeginFrame/EndFrame의 세마포어 2개 + 펜스 1개 조합 acquire→submit→present, OUT_OF_DATE/ SUBOPTIMAL 스왑체인 리사이즈 처리까지 구현되어 있습니다.
여기부터는 아직입니다: 실제 리소스 생성 · 드로우 경로.
InitializeVertexLayout/Texture/Sampler/RenderTarget/RenderPass, 모든 Update*/Set*/Unset*, DispatchCompute, DrawIndexPrimitive, Clear가 전부 빈 함수입니다. InitializeBuffer는 편집 중인 상태 그대로 남아 있어 지금 시점에는 컴파일도 되지 않습니다.
VulkanMemory: 4MB 블록 + 블록 내부는 TLSFArena로 서브 할당.
Vulkan 메모리 타입별로 고정 4MB VkDeviceMemory 블록을 필요할 때 만들고 비면 반환하는 구조 — 리소스마다 VkDeviceMemory를 따로 할당하는 흔한 안티패턴을 피했습니다. 블록 내부의 실제 서브 할당은 Container 라이브러리의 TLSF 구현을 그대로 재사용합니다.

Asset

3단계, 3개의 서로 다른 스레드 — Parse가 메인 루프를 절대 막지 않고, Upload는 RHIWorker 밖에서 절대 실행되지 않습니다. 

eNone --TaskWorker.Parse()--> eLoaded --AssetCommander.Load() GPU 업로드--> (상태 전이 없음)
Parse = CPU 바운드, TaskWorker 풀(유휴 워커가 있을 때만 — 없으면 AssetSystem 큐에 다시 넣어 다음 패스로 미룸)
Load  = GPU 업로드(AssetCommander), RHIWorker 전용
※ eAssetState는 eNone/eError/eLoaded/eExpried(오타)/eDirty뿐 — "완료(eReady)" 상태는 존재하지 않고,
  준비 여부는 texture/buffer/shader 포인터가 non-null인지로 판단
04

실제 코드에서 발견한 것들

설계 문서가 아니라 소스를 직접 읽고 확인한 내용

이 페이지는 Docs/Design_*.md 요약이 아니라, 이번에 Engine/Private 과 Engine/Public의 실제 구현을 다시 읽으면서 확인한 내용입니다. 문서와 다른 부분, 그리고 흥미로운 설계 판단 몇 가지를 그대로 남겨둡니다.

LightingPass는 블렌드를 켜지 않아서, 여러 라이트가 있으면 마지막 라이트만 남습니다.
라이트 하나당 풀스크린 쿼드 Draw 한 번(Directional/Point/Spot 셰이더 분리)인데, GetPipeLine()이 채우는 RHIPipeLineCreateDesc에 blend 필드가 끝까지 비어 있습니다. 로컬로 RHIBlendState blendState; blendState.enable = false;를 선언은 하지만 어디에도 대입·전달되지 않는 죽은 코드입니다 — 이 구조가 성립하려면 blend를 꺼야 하는 게 아니라 라이트별로 누적되도록 additive 블렌딩을 켜야 합니다. 라이트가 하나뿐인 데모에서는 드러나지 않는 버그입니다.
RenderGraph::Execute()는 Pass 하나만 준비 안 돼도 그 프레임 전체를 중단합니다.
정렬된 순서를 돌다가 어떤 Pass의 GetResourceState() != eReady 이거나 Draw()가 false를 반환하면 그 자리에서 루프를 break — 문제 있는 Pass 하나만 건너뛰는 게 아니라 이후 Pass 전부가 이번 프레임에 안 그려집니다.
WorldCommandList는 사실상 미사용 상태입니다.
struct WorldCommand {}; 정도만 있고 WorldContext::Prepare() 는 그냥 // TODO. World→Renderer 트래픽은 전부 Commander/RenderCommandList로만 흐르고 있어서, 이 클래스는 설계 초기 흔적으로 보입니다.
RHICommand 실제 개수는 41개입니다.
기존 설계 문서의 "45개 이상"이라는 서술은 실측과 다릅니다 — 초기화 11 · 갱신 3 · 리사이즈 2 · 제거 7 · Set 8 · Unset 7 · 프레임/디스패치 3.
RenderCommandList::ExecuteAll()에 기술적으로는 UB인 패턴이 하나 있습니다.
태스크를 실행한 뒤 명시적으로 execute->~RenderTask()를 호출하고, 바로 다음 줄에서 execute->next를 읽습니다. next가 trivial한 raw pointer라 실제로는 문제없이 동작하지만, 소멸자 호출 후 멤버를 읽는 건 엄밀히는 정의되지 않은 동작입니다.
Asset 파싱은 항상 병렬은 아니고, 기회적(opportunistic)입니다.
유휴 TaskWorker가 있을 때만 별도 스레드로 넘기고, 없으면 AssetSystem::AddTask()로 다시 큐에 넣어 다음 패스로 미룹니다 — 동기적으로 직접 파싱하지 않고, "병렬 파싱"이 항상 보장되는 것도 아닙니다.
Application/Engine 초기화 중 AssetWorker가 두 번 만들어집니다.
Engine::InitAsset()이 AssetWorker를 하나 만든 직후, InitWorker()가 SetTaskThread(4) 설정으로 또 하나를 만들어 m_assetWorker를 덮어씁니다 — 첫 번째 인스턴스는 조용히 버려집니다. WorldCommandList와 같은 종류의 흔적성 코드로 보입니다.
Win32Window의 WM_CLOSE 처리와 실제 종료 경로는 다릅니다.
WndProc의 switch 문에 케이스 사이 break가 없어서, WM_CLOSE에서 설정한 m_status = eClosed가 같은 호출 안에서 곧바로 다시 eActive로 덮어써집니다. 실제 종료는 상태 체크가 아니라 close 콜백 → Engine::Shutdown() → m_window 리셋 → Run() 루프의 null 체크로 빠져나가는 경로로 이루어집니다.
InputStorage는 더블 버퍼가 아니라 락프리 트리플 버퍼입니다.
m_InputData[3] — 메인 스레드가 쓰는 m_CurrData, 핸드오프용 atomic<InputData*> m_PrevData, WorldWorker가 읽는 m_LogicData. 256개 키코드 전체 상태를 4비트(Down/Press/Release/Repeat)로 패킹한 128바이트 배열로 저장합니다.
TaskWorker "풀"은 사실 AssetWorker 전용이고, 로드밸런싱이 없습니다.
TaskWorker 자체는 작업 하나짜리 워커일 뿐 — "풀"은 AssetWorker가 가진 DynamicArray<RefPtr<TaskWorker>>이고, 스레드 개수 4는 Engine.cpp에 하드코딩. 분배는 라운드로빈이 아니라 첫 번째로 유휴 상태인 워커를 선형 탐색. Win32InputHandler의 입력 큐도 backpressure 없이 MAX_QUEUE_SIZE를 넘으면 그냥 통째로 비워 밀린 입력을 버립니다.
제네릭 Task<Args...>가 실제로는 두 가지 다른 방식으로 쓰입니다.
Task<Args...>는 std::function<void(Args...)> 얇은 래퍼일 뿐인데, Asset/TaskWorker 쪽은 DefaultTask = Task<>로 인자 없이 클로저에 전부 캡처해서 쓰고, RenderTask : Task<Renderer*, RefPtr<RHICommandList>> 는 실제 인자를 받으며 커스텀 할당기 위에 인트루시브 리스트 노드로 placement-new됩니다 — "제네릭" 템플릿이 사실상 한쪽 전용 소비자를 위해 존재하는 셈입니다.
Asset 상태 머신에는 "완료(eReady)" 상태가 없습니다.
실제 eAssetState는 eNone / eError / eLoaded / eExpried(오타) / eDirty 뿐입니다. GPU 업로드(AssetCommander::Load) 이후 별도 상태 전이가 없어서, 준비 여부는 상태값이 아니라 texture/buffer/shader 포인터가 non-null인지로 다른 코드가 추론합니다.
RHIDrawIndexDesc / RHIDispatchDesc가 자기 태그 대신 남의 태그를 씁니다.
eResourceType은 9개 값(Buffer/Texture/Sampler/Shader/PipeLine/ Layout/Target/Const/Pass)이 따로 있는데, RHIDrawIndexDesc는 eLayout을, RHIDispatchDesc는 ePipeLine을 그대로 재사용합니다 — Draw 커맨드와 VertexLayout이 desc.type 값만으로는 구분되지 않습니다.
PNGParser에는 실제로 타입이 어긋나는 버그가 있습니다.
stbi_load_from_memory는 항상 8비트 데이터를 반환하는데, componentType은 소스에 따라 eFloat/ eUInt로 설정됩니다 — 둘 다 실제 eUByte와 맞지 않습니다. sRGB 판별도 IHDR 바로 다음 청크 하나만 검사해서(스펙상 IDAT 전 어디든 올 수 있음) 코드 안 TODO로 부정확함이 남아 있습니다.

해결한 이슈

저장소 이슈 트래커와 디버그 로그 기준

섀도우 acne / NaN / 깜빡임 해결

증상
카메라 회전 중 간헐적으로 물체 전체가 검게 표시됨. RenderDoc 단일 프레임 캡처에서는 정상.
원인
grazing 각도에서 RPDB depth-bias의 분모(det)가 0에 가까워지며 NaN이 발생, FragColor까지 그대로 전파됨.
해결
NaN을 사후에 가리려 하지 않고 발생 자체를 차단(safeDet = mix(EPSILON, det, step(EPSILON, abs(det)))) — 곱셈이나 mix로는 NaN을 지울 수 없기 때문. grazing 컷오프도 맨 끝에서 한 번만 곱하도록 이동(dFdx 앞에서 분기하면 GLSL에서 undefined behavior).

Multi-proxy dirty flag race 해결

증상
같은 Transform을 공유하는 라이트와 메시를 가진 엔티티가, 첫 프레임 이후 둘 중 하나의 업데이트를 조용히 멈춤.
원인
Hash Storm 최적화의 if (!IsDirty()) OnUpdate() 가드가 공유 Component에 있던 dirty 플래그를 지워버림 — 먼저 처리된 Proxy가 둘 다의 플래그를 지운 셈.
해결
Proxy별 dirty 세대(generation) 카운터 도입: Component는 변경 시 세대 번호만 증가시키고, 각 Proxy가 자신이 마지막으로 본 세대를 독립적으로 추적.

Point Light 입사 방향 아티팩트 미해결

증상
표면 아래에 위치한 라이트가, 라이트 볼륨 경계 부근에서 오히려 표면을 더 밝게 만듦.
원인
Two-sided 라이팅 보정의 max(dot(normal, rayDir), 0.0)가 이 경우 0으로 클램프되어, 입사 방향 보정이 아예 걸리지 않음.
개선 방향
보정을 fragment→light가 아니라 light→fragment 방향으로 계산. 섀도우 맵이 도입되면 이 케이스는 자연스럽게 가려지기도 함.

성능 최적화 여정 — 100만 인스턴스 병목 추적기

이슈 #25 한 건에서 한 달(2026-04-22 ~ 2026-05-23)에 걸쳐 실제로 있었던 일입니다 — 프로파일러로 병목을 찾고, 고치고, 중간에 리팩토링으로 오히려 성능이 떨어지기도 하고, 다시 고치고, 다음 병목을 찾는 과정을 그대로 옮겼습니다.

1) 3만 인스턴스 회전 — Render Thread가 scalar GLM 행렬 곱에 발목잡힘.
플레임그래프에서 SceneProxy::UpdateRotation → UpdateTransform → glm::detail::mul4x4가 압도적 1위. GLM이 SIMD 없이 컴파일돼서 mat4×mat4가 전부 스칼라 루프로 도는 게 원인이었습니다.
2) 리팩토링 중 오히려 후퇴 — ISM 도입 과정에서 성능이 10배 나빠짐.
Unreal의 FInstancedStaticMeshSceneProxy를 참고해 InstancedStaticMeshProxy 개념을 도입하던 중, 이전 엔진이 100만 개 @ 20 FPS(Release)를 버텼던 반면 리팩토링 직후 버전은 10만 개에서야 20 FPS로 오히려 후퇴했습니다 — 리팩토링이 항상 개선을 보장하지 않는다는 걸 프로파일러가 바로 알려준 사례입니다.
3) TransformPass로 회복 + 그 이상 — TRS 행렬 계산을 GPU 컴퓨트 셰이더로 완전히 이전.
기존에는 MeshBatch 안에서 인스턴스마다 CPU로 Translation×Rotation×Scale 행렬을 만들었는데, 이제 local_size_x = 64 컴퓨트 셰이더가 RawTransform SSBO(위치·쿼터니언·스케일)를 읽어 Transform SSBO(mat4)로 계산합니다. 결과: Debug 10만 개 안정 20 FPS(리팩토링 전에는 3만 개에서도 10 FPS 밑으로 떨어졌음), Release 50만 개 ~15 FPS.
4) 다음 병목까지 찾아둠 — HashTable::Rehash의 임시 배열이 프록시 등록 시 힙 할당을 유발.
행렬 계산 병목이 풀리자 프로파일러가 다음으로 HashTable::Rehash 내부의 임시 DynamicArray 생성을 새 할당 핫스팟으로 지목했습니다. 다음 단계로 대량 삽입 전에 예상 용량만큼 Reserve를 미리 호출해서 핫 패스에서 Rehash 자체를 없애는 방향을 이슈에 남겨뒀습니다.
Hash Storm: 병목은 "무조건 실행되는 삽입"이었습니다.
100만 인스턴스에서 프레임당 200만 번 호출되던 HashSet::Insert를 if (!IsDirty()) 가드로 프레임당 1회로 축소: 5.4 → 22 FPS Debug.
