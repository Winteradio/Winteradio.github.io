Custom C++ Game Engine

A 4-thread engine — World · Render · RHI · Asset — that mirrors Unreal's separation of game logic from rendering. Everything outside the day job's OpenGL ES map renderer lives here, built on four systems written from scratch (see the project pages in the nav above).

Threading & Frame Synchronization

how the 4 workers hand a frame to each other

FrameGate is a producer-consumer condition variable pair: World signals it once ECS update is done, Renderer waits on it, then signals its own gate for RHI. Nobody polls; nobody spins.

RenderCommandList is zero-allocation: two LinearArenas ping-pong — World writes RenderTask lambdas into one while Render drains the other, so a frame boundary never costs a heap allocation.

MeshBatch owns exactly one MeshDrawCommand per (mesh, material) pair.
Two SSBOs live on it — rawTransform (raw position / quaternion / scale per instance) and transform (the computed mat4s TransformPass writes) — plus the vertex layout and a material reference.
GeometryPass reads one MeshDrawCommand, issues one draw call.
Bind vertex layout, bind the transform SSBO, bind material — a single indexed draw covers every instance in the batch. This is the mechanism behind "1 draw call per batch."
RenderPass splits into two abstract bases, not one.
ComputePass just dispatches a compute shader — used by TransformPass (GPU TRS) and CullingPass (currently a pass-through stub). GraphicPass owns full pipeline state — clear, depth, blend, rasterizer — used by GeometryPass and LightingPass.
RenderGraph is just an ordered list of RenderPass.
Execute() walks it once a frame: TransformPass → GeometryPass → LightingPass ×N lights. Adding a pass means appending to the list.

what GeometryPass writes, before LightingPass composites them

03

What Changed When It Was Measured

shadow mapping · deferred rendering · 1M-instance optimization
TransformPass moved TRS→matrix onto the GPU.
A compute shader (local_size_x=64) replaced per-instance CPU GLM math. 500K instances @ 20 FPS Release — roughly 10× over the CPU path, which couldn't hold 10 FPS Debug past 30K instances.
Hash Storm: an unconditional insert was the real bottleneck.
Every instance mutation queued a HashSet::Insert — 2,000,000 calls/frame at 1M instances, 15–20% of frame time. An if (!IsDirty()) guard cut that to once per frame regardless of mutation count: 5.4 → 22 FPS Debug, 185ms → 45ms/frame.
Shadow shimmering — fixed by snapping, not by more samples.
Camera-frustum center quantized to the shadow map's texel grid before recomputing the light view matrix, using a bounding-sphere frustum. Removes shadow jitter under camera motion without extra PCF taps.

the 1M-cube capture that the Hash Storm numbers above refer to is 112MB — over GitHub's 100MB per-file push limit, so the 500K-cube capture (same technique, same code path) stands in here; the real one is one click away on the right.
