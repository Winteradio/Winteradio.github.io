Renderer/LightingPass.cpp:64-65, 160-185 라이트가 여러 개면 마지막 라이트만 남는 블렌딩 버그 버그 · 미수정
- 증상: 라이트 하나당 풀스크린 쿼드 Draw 한 번씩 그리는 구조인데, 여러 라이트가 있으면 화면에는 가장 마지막에 그려진 라이트의 결과만 남아야 정상적으로 보임(누적이 안 됨).
- 원인: GetPipeLine()이 채우는 RHIPipeLineCreateDesc의 blend 필드가 끝까지 비어 있음. 로컬로 RHIBlendState blendState; blendState.enable = false;를 선언은 하지만 어디에도 대입·전달되지 않는 죽은 코드.
- 개선 방향: additive 블렌딩(ONE, ONE)을 desc.blend에 연결해서 라이트 기여도가 프레임버퍼에 누적되도록 해야 함. 라이트가 하나뿐인 데모/스크린샷에서는 드러나지 않아서 지금까지 남아있었던 것으로 보임.
Framework/Application.cpp, Engine.cpp AssetWorker가 초기화 중 두 번 생성되고 첫 번째는 버려짐 버그 · 미수정
- 원인: Engine::InitAsset()이 AssetWorker를 하나 만든 직후, InitWorker()가 SetTaskThread(4) 설정으로 또 하나를 만들어 m_assetWorker를 덮어씀.
- 영향: 기능상 즉시 문제를 일으키진 않지만(두 번째 인스턴스가 정상 동작), 첫 번째 인스턴스가 만든 스레드/리소스가 조용히 새어나가는 초기화 순서 버그.
Platform/Win32/Win32Window.cpp — WndProc WM_CLOSE에서 설정한 종료 상태가 같은 호출 안에서 즉시 덮어써짐 버그 · 미수정
- 원인: WndProc의 switch 문에 케이스 사이 break가 없음 — WM_CLOSE에서 m_status = eClosed를 설정한 직후 아래 케이스로 흘러들어가(fallthrough) 다시 eActive로 되돌아감.
- 실제 동작: 겉보기 상태 머신은 신뢰할 수 없고, 실제 종료는 close 콜백 → Engine::Shutdown() → m_window 리셋 → Run() 루프의 null 체크로 빠져나가는 별개 경로로 이루어짐 — 결과적으로 종료 자체는 되지만, GetStatus()를 신뢰하는 코드가 있다면 오작동함.
RHI/RHIDescriptions.h:397, 410 RHIDrawIndexDesc / RHIDispatchDesc가 자기 타입 태그 대신 다른 타입 태그를 재사용 버그 · 미수정
- 원인: eResourceType은 Buffer/Texture/Sampler/Shader/PipeLine/Layout/ Target/Const/Pass 9개 값이 따로 있는데, RHIDrawIndexDesc는 자기 것 대신 eLayout을, RHIDispatchDesc는 ePipeLine을 그대로 재사용.
- 영향: desc.type 값만으로는 Draw 커맨드와 VertexLayout, Dispatch 커맨드와 PipeLine을 구분할 수 없음 — 현재는 타입을 이 필드로 분기하는 코드가 없어서 조용히 묻혀 있음.
Asset/PNGParser.cpp:66-73 PNGParser의 componentType이 실제 데이터 타입과 어긋남 버그 · 미수정
- 원인: stbi_load_from_memory는 항상 8비트 데이터를 반환하는데, componentType은 소스에 따라 eFloat/ eUInt로 설정됨 — 둘 다 실제 eUByte와 맞지 않음.
- 부수적으로: sRGB 판별(IsSRGB)이 IHDR 바로 다음 청크 하나만 검사 — PNG 스펙상 sRGB 청크는 IDAT 이전 어디에나 올 수 있어서 오탐/누락 가능(코드 안에 TODO로 남아있음).
Renderer/RenderCommandList.cpp — ExecuteAll() 소멸자 호출 후 멤버를 읽는 기술적 UB 패턴 버그(경미) · 미수정
- 원인: 태스크를 실행한 뒤 명시적으로 execute->~RenderTask()를 호출하고, 바로 다음 줄에서 execute->next를 읽음.
- 실제 위험도: next가 trivial한 raw pointer라 지금 컴파일러/최적화 수준에서는 문제없이 동작하지만, 엄밀히는 정의되지 않은 동작 — 공격적인 최적화나 sanitizer 환경에서 깨질 수 있음.

버그는 아니지만 — 미구현 / 죽은 코드

Renderer/RenderPass/CullingPass.cpp, culling.cs.glsl CullingPass는 리소스 배선만 되어 있는 스텁 미구현
- Draw()는 상태 체크만 하고 항상 true 반환, GetResourceState()도 항상 eReady로 하드코딩.
- 카메라/인다이렉트 드로우 버퍼/가시성 버퍼/AABB 버퍼를 바인딩하는 SetCommand()는 이미 있어서, 나중에 실제 컬링 로직만 끼워넣을 수 있는 상태.
- culling.cs.glsl은 이름과 달리 실제로는 compute가 아니라 vertex 셰이더 내용 — 복사해온 자리표시자로 보임.
Renderer/RenderPass/OverlayPass.h, .cpp OverlayPass는 완전히 빈 파일 미구현
- 두 파일 다 0바이트, 클래스 선언조차 없음 — 시작조차 안 된 자리표시자.
World/WorldContext.cpp — WorldCommand WorldCommandList는 사실상 미사용 흔적 코드
- struct WorldCommand {}; 정도만 있고 WorldContext::Prepare()는 그냥 // TODO. World→Renderer 트래픽은 전부 Commander/RenderCommandList로만 흐르고 있어서, 설계 초기 흔적으로 보임.
Asset/AssetTypes.h:47-54 Asset 상태 enum에 "완료(eReady)" 상태가 없고, 오타(eExpried)가 있음 설계 노트
- 실제 eAssetState는 eNone / eError / eLoaded / eExpried(오타, Expired가 맞음) / eDirty뿐. GPU 업로드 이후 별도 상태 전이가 없어서, 준비 여부는 상태값이 아니라 texture/buffer/shader 포인터가 non-null인지로 다른 코드가 추론함.
