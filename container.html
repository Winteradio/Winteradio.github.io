02

How It Actually Runs

Robin Hood hashing and the TLSF bitmap search, mechanically

Robin Hood Hashing

Slot layout
{ Data data; int16_t psl; }, default psl = -1 — that's the empty-slot sentinel. No separate tombstone flag exists anywhere.
Insert = linear probe + steal-from-the-rich
Walk forward one slot at a time, incrementing the entrant's PSL each step. On landing in an occupied slot: if (newSlot.psl > oldSlot.psl) swap(newSlot, oldSlot) — the entrant only displaces a resident whose PSL is strictly lower (closer to its own ideal slot). The displaced resident keeps probing forward with its PSL still climbing.
Growth
Rehashes at 70% load factor, doubling capacity and re-running the identical insert-with-swap into a fresh array.
Erase = backward-shift, not tombstoning
The freed slot is marked psl = -1, then a walk pulls every later element in that probe chain backward by its own PSL (shrinking it accordingly) until hitting a slot that's already home or empty. The hole physically dissolves instead of leaving a marker behind — which is exactly what keeps Find()'s early-exit rule (stop once probe distance exceeds a slot's PSL) valid forever, with no stale tombstones ever accumulating.

TLSFArena — the FL/SL bitmap search

Two-level classification
FL index = most-significant-bit position of the size (64 rows). SL index = a 4-bit linear subdivision within that power-of-two range (16 columns). bucketIndex = fl*16 + sl indexes a flat free-list-head array.
The search itself is branch-free
One uint64_t bitmap per FL row, one bit per SL column. Allocate() masks off every class smaller than what's needed (bitmap[fl] & (~0 << sl)) and takes the lowest set bit of what's left — the nearest sufficiently-large free class, found with pure bit ops, never touching the block list itself.
Free() merge direction
Only merges two blocks that are both free and physically adjacent — adjacency is decided by comparing byte offset, never by array index. The absorbed block's slot is recycled onto a free-index list rather than shrinking the block array.
Why two link pairs per block
prev/next track physical byte-order neighbors and stay valid whether a block is free or allocated — it's the only way Free() can find what's physically touching a block. prevFree/nextFree track the same-size-class free-list chain and are only meaningful while free. They encode two unrelated relationships on the same block; collapsing them into one pair would break either merging or size-class lookup.

Arena vs. LinearArena

Arena: individually freeable
Every allocation is its own ::operator new with a Page header linked into a doubly-linked list — one pointer maps 1:1 to one OS allocation and one list node, so Deallocate(ptr) can walk the list and free exactly that one.
LinearArena: bulk-only by construction
Allocations just bump an offset within a shared page (new pages added on overflow). Nothing records which live objects share a page, so there's no way to free just one — only Reset(), which zeroes every page's offset at once.

Real Issues

#25 Use-After-Free in DynamicArray Move Constructor closed
- Cause: the move constructor moved m_data but not m_allocator — the moved-from object kept a live, default-constructed allocator with no real claim on the transferred memory, so its destructor freed memory it no longer owned.
- Fix: move the allocator in the initializer list alongside the data; same fix ported to move-assignment.
#24 HashMap::operator[] Reference Invalidation during Rehash open
- Cause: m_map[A] = m_map[B] takes the RHS reference first; the LHS access can then trigger a rehash that allocates a new backing array and discards the old one, dangling the RHS.
- Direction: logged as an inherent trade-off of open addressing (the same cache-locality property that makes lookups fast), not yet patched — std::unordered_map doesn't hit this because its nodes never move.

From building TLSFArena directly

Not GitHub tickets — bugs caught during development, worth keeping because of what they reveal about the design.

Merge direction decided by array index instead of byte offset
- Cause: isLeft was computed by comparing block indices in the block array, which have no relationship to physical memory position. Two blocks adjacent in offset can sit anywhere in the array.
- Fix: compare offset directly — the same rule the confirmed-current code above still follows.
Bitmap bit cleared unconditionally on every free-list removal
- Cause: removing a block from its size-class free-list cleared that class's bitmap bit even when other free blocks remained in the same bucket, making the allocator blind to blocks that were still there.
- Fix: only clear the bit when the removed block was the bucket's sole occupant.
A merge result silently discarded
- Cause: Free() calls Merge() twice (once per physical neighbor) but stored the result with = instead of |= — a successful left-merge could be overwritten by a failed right-merge, then the block got re-inserted into a free-list it was already merged out of.
- Fix: accumulate both results with |= before deciding whether to insert.
