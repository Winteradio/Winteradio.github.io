STL-free Container

DynamicArray, StaticArray, HashMap/HashSet(Robin Hood 해싱), 그리고 그 아래를 받치는 할당자들 — Arena, LinearArena, TLSFArena. TLSFArena는 임의 바이트 범위에서 O(1) 할당/해제/분할/병합이 가능합니다.

01

핵심 내용

Robin Hood 해싱은 "가난한 쪽이 이긴다"는 규칙 하나로 동작합니다.
삽입할 때 자기보다 PSL(제자리에서 얼마나 멀어졌는지)이 낮은, 즉 "부자"인 원소만 밀어냅니다. 밀려난 원소는 계속 앞으로 탐색하며 자기 PSL을 올립니다.
삭제는 tombstone 없이 backward-shift로 처리합니다.
빈 슬롯이 생기면 뒤따르는 원소들을 자기 PSL만큼 앞으로 당겨서 구멍을 물리적으로 없애버립니다 — Find()의 조기 종료 조건이 영원히 유효하게 유지되는 이유입니다.
TLSFArena의 탐색은 분기 없이 비트 연산만으로 끝납니다.
필요한 크기보다 작은 클래스를 비트마스크로 지워버리고, 남은 것 중 최하위 비트를 찾으면 그게 가장 가까운 free 블록입니다. 블록 리스트를 훑을 필요가 없습니다.

Robin Hood 해싱

슬롯 구조
{ Data data; int16_t psl; }, 기본값 psl = -1 — 이게 빈 슬롯을 나타내는 sentinel입니다. 별도 tombstone 플래그는 어디에도 없습니다.
삽입 = 선형 탐사 + "부자" 밀어내기
한 칸씩 나아가며 신규 원소의 PSL을 올립니다. 점유된 슬롯에 도달하면 if (newSlot.psl > oldSlot.psl) swap(newSlot, oldSlot) — 신규 원소는 자기보다 PSL이 낮은(제자리에 더 가까운) 원소만 밀어냅니다. 밀려난 원소는 계속 앞으로 탐사합니다.
증설
load factor 70%에서 리해시하며 용량을 두 배로 늘리고, 같은 삽입+swap 로직을 새 배열에 다시 돌립니다.
삭제 = backward-shift, tombstone 없음
비운 슬롯은 psl = -1로 표시하고, 같은 탐사 체인의 뒤따르는 원소들을 자기 PSL만큼 뒤에서 앞으로 당겨오며 PSL을 줄입니다 — 제자리이거나 빈 슬롯을 만날 때까지 반복. 구멍을 표시만 하는 게 아니라 물리적으로 없애버려서, Find()의 조기 종료 규칙(탐사 거리가 슬롯의 PSL을 넘으면 중단)이 영원히 유효합니다.

TLSFArena — FL/SL 비트맵 탐색

2단계 분류
FL 인덱스 = 크기의 최상위 비트 위치(64행). SL 인덱스 = 그 2의 거듭제곱 범위 안에서의 4비트 선형 세분(16열). bucketIndex = fl*16 + sl이 평평한 free-list 헤드 배열을 가리킵니다.
탐색 자체는 분기가 없습니다
FL 행마다 uint64_t 비트맵 하나, SL 열마다 비트 하나. Allocate()는 필요한 크기보다 작은 클래스를 전부 마스킹해서 지우고 (bitmap[fl] & (~0 << sl)), 남은 것 중 최하위 비트를 취합니다 — 블록 리스트를 전혀 훑지 않고 순수 비트 연산만으로 가장 가까운 free 클래스를 찾습니다.
Free() 병합 방향
물리적으로 인접하면서 둘 다 free인 블록만 병합합니다 — 인접 여부는 배열 인덱스가 아니라 바이트 offset 비교로 결정됩니다. 흡수된 블록의 슬롯은 블록 배열을 줄이는 대신 free-index 목록으로 재활용됩니다.
블록마다 링크 쌍이 두 개인 이유
prev/next는 물리적 바이트 순서상의 이웃을 추적하며 free든 allocated든 항상 유효합니다 — Free()가 물리적으로 맞닿은 블록을 찾는 유일한 방법입니다. prevFree/nextFree는 같은 크기 클래스의 free-list 체인을 추적하며 free일 때만 의미가 있습니다. 같은 블록의 서로 무관한 두 관계를 나타내기 때문에, 하나로 합치면 병합이나 크기 클래스 조회 둘 중 하나가 깨집니다.

Arena vs. LinearArena

Arena: 개별 해제 가능
모든 할당이 각자 ::operator new이고 Page 헤더가 이중 연결 리스트에 연결됩니다 — 포인터 하나가 OS 할당 하나, 리스트 노드 하나에 정확히 대응하므로 Deallocate(ptr)가 리스트를 훑어 그 하나만 정확히 해제할 수 있습니다.
LinearArena: 처음부터 일괄 해제 전용
할당은 공유 페이지 안에서 offset을 밀기만 합니다(넘치면 새 페이지 추가). 어떤 페이지를 누가 쓰고 있는지 기록이 없어서 하나만 골라 해제할 방법이 없고, 페이지 전체 offset을 한 번에 초기화하는 Reset()만 존재합니다.

결과물 / 이슈

#25 Use-After-Free in DynamicArray Move Constructor 해결
- 원인: 이동 생성자가 m_data는 옮겼지만 m_allocator 는 옮기지 않아서, 이동된 원본 객체가 더 이상 소유하지 않는 메모리를 소멸자에서 해제해버렸습니다.
- 해결: 초기화 리스트에서 할당자도 함께 이동. move-assignment에도 동일하게 적용.
#24 HashMap::operator[] Reference Invalidation during Rehash 미해결
- 원인: m_map[A] = m_map[B]에서 RHS 참조를 먼저 얻는데, LHS 접근이 리해시를 유발하면 기존 배열이 해제되면서 RHS 참조가 댕글링됩니다.
- 개선 방향: 연속 메모리 open addressing이라 캐시 지역성을 얻는 대신 생기는 근본적인 트레이드오프로 기록해두고 아직 패치하지 않았습니다 — std::unordered_map은 노드가 이동하지 않아서 이 문제가 없습니다.

TLSFArena를 직접 만들면서 잡은 버그

GitHub 이슈 티켓은 아니지만, 개발 과정에서 발견하고 고친 것들 — 설계가 왜 그렇게 됐는지 보여줘서 남겨둡니다.

병합 방향을 배열 인덱스로 판단하던 버그
- 원인: isLeft를 블록 배열 상의 인덱스로 비교했는데, 인덱스는 물리적 메모리 위치와 아무 관계가 없습니다. offset이 인접한 두 블록이 배열 어디에나 있을 수 있습니다.
- 해결: offset을 직접 비교하도록 수정 — 위 설명의 현재 코드가 따르는 규칙 그대로입니다.
free-list에서 제거할 때마다 비트맵 비트를 무조건 지우던 버그
- 원인: 같은 버킷에 다른 free 블록이 남아있어도 비트맵 비트를 지워버려서, 할당자가 여전히 존재하는 블록을 못 찾게 됨.
- 해결: 제거된 블록이 그 버킷의 유일한 원소였을 때만 비트를 지우도록 수정.
병합 결과가 조용히 덮어써지던 버그
- 원인: Free()가 물리적 이웃마다 한 번씩, Merge()를 두 번 호출하는데 결과를 =로 저장 — 왼쪽 병합 성공이 오른쪽 병합 실패로 덮어써지면서, 이미 병합되어 빠진 블록을 다시 free-list에 삽입하려 시도.
- 해결: 두 결과를 |=로 누적한 뒤 삽입 여부를 판단.
